# Аксиомы проекта (Core Beliefs)

> Это фундаментальные принципы, определяющие «вкус» проекта.
> Сюда попадают ТОЛЬКО правила, которые **невозможно проверить линтером или тестом**.
> Всё, что можно автоматизировать, должно быть в линтере, тесте или скрипте — не здесь.
>
> Этот файл входит в иммутабельное ядро (см. `docs/index.md`, секция B.1).

## Как читать этот документ

Каждая аксиома имеет формат:
- **Утверждение** — что мы делаем (или не делаем).
- **Обоснование** — почему.
- **Пример** — как это выглядит на практике.

---

## 1. Репозиторий — единственный источник истины

**Утверждение:** Всё, что нужно для понимания проекта, должно лежать в репозитории. Решения, принятые в чатах, звонках или головах людей — не существуют для агентов и новых участников.

**Обоснование:** Агент видит только то, что лежит в контексте. Знание, которого нет в репо — это знание, которого нет.

**Пример:** После обсуждения архитектурного решения в Slack — зафиксируй итог в Design Doc.

## 2. Предпочитаем дублирование сложной абстракции

**Утверждение:** Если два модуля нуждаются в похожей логике, допустимо продублировать код, а не создавать сложную абстракцию «на вырост».

**Обоснование:** Агенты легко создают код, но плохо поддерживают хрупкие абстракции. Дублирование проще читать и менять, чем глубокую иерархию наследования.

**Пример:** Два API-эндпоинта с похожей валидацией — каждый валидирует у себя, без общего абстрактного валидатора.

## 3. Интерфейсы должны быть плоскими

**Утверждение:** Структуры данных, API-контракты и конфигурации должны иметь минимальную вложенность. Предпочитаем «широкие и плоские» объекты «узким и глубоким».

**Обоснование:** Плоские структуры легче парсить, типизировать и отлаживать — как человеку, так и агенту.

**Пример:** `{ user_id, user_name, user_email }` вместо `{ user: { profile: { contact: { email } } } }`.

## 4. Ошибки — структурированные данные, а не строки

**Утверждение:** Все ошибки логируем структурированно (JSON). Текстовые логи вида `console.log("error happened")` недопустимы.

**Обоснование:** Структурированные логи можно парсить, фильтровать и анализировать — как агентами, так и инструментами мониторинга.

## 5. Маленькие PR лучше больших

**Утверждение:** Одна задача = один Pull Request. PR более чем на 400 строк (без учёта автогенерации) — повод для разделения.

**Обоснование:** Маленькие PR легче ревьюить (и человеку, и агенту), быстрее мерджатся и генерируют меньше конфликтов при параллельной работе.

---

> **Как добавить новую аксиому:** Убедись, что правило действительно нелинтируемое.
> Если его можно проверить автоматически — добавь проверку в `scripts/verify.sh` или конфиг линтера.
> Каждая аксиома должна иметь Утверждение, Обоснование и Пример.
