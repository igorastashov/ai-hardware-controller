# Навигатор репозитория

> **Это главный документ проекта.** Сюда приходят и люди (из `README.md`), и AI-агенты (из `AGENTS.md`).
> Прочитай этот файл целиком перед началом любой работы.

---

## Часть A: Карта территории

```
/
├── README.md                    — Точка входа для человека (что за проект, как запустить)
├── AGENTS.md                    — Точка входа для AI-агента (системный промпт)
├── ARCHITECTURE.md              — Верхнеуровневая архитектура: слои, границы, зависимости
│
├── docs/                        — Система знаний проекта
│   ├── index.md                 — ★ ВЫ ЗДЕСЬ. Главный навигатор.
│   │
│   ├── design-docs/             — Фундаментальные решения (меняются редко)
│   │   ├── index.md             — Каталог всех Design Docs со статусами
│   │   └── core-beliefs.md      — Аксиомы проекта: нелинтируемые принципы «вкуса»
│   │
│   ├── exec-plans/              — Память проекта (Context Graph / Decision Traces)
│   │   ├── TEMPLATE.md          — Обязательный шаблон для новых планов
│   │   ├── active/              — Планы в работе (один план = одна ветка = один агент)
│   │   ├── completed/           — Архив завершённых планов (ТОЛЬКО для поиска!)
│   │   └── tech-debt-tracker.md — Единый реестр технического долга
│   │
│   ├── product-specs/           — Продуктовые требования (что делаем и для кого)
│   │   └── index.md             — Каталог спецификаций
│   │
│   └── references/              — Шпаргалки по стеку и инструментам для агентов
│       └── index.md             — Каталог справочных материалов
│
├── scripts/                     — Engineering Harness (инструменты проверки)
│   ├── index.md                 — Контракты и описания всех скриптов
│   └── verify.sh                — Главный скрипт валидации (линт, типы, тесты)
│
├── src/                         — Исходный код (создаётся при старте разработки)
│
└── .cursor/
    └── rules/
        └── repo-conventions.mdc — Правила, автоматически загружаемые в Cursor IDE
```

## Часть B: Правила ведения документации (Meta-Guidelines)

### B.1. Иммутабельное ядро (НЕ УДАЛЯТЬ, НЕ ПЕРЕИМЕНОВЫВАТЬ)

Следующие файлы и папки составляют **каркас репозитория**. Они не могут быть удалены, переименованы или перемещены ни человеком, ни агентом. Их содержание может обновляться, но структура — постоянна.

| Файл / Папка | Почему неприкосновенен |
|---|---|
| `docs/index.md` | Главный навигатор. Без него агент теряет карту. |
| `AGENTS.md` | Точка входа агента. Без него агент работает вслепую. |
| `README.md` | Точка входа человека. |
| `ARCHITECTURE.md` | Карта архитектуры. |
| `docs/design-docs/core-beliefs.md` | Аксиомы, определяющие «вкус» проекта. |
| `docs/exec-plans/TEMPLATE.md` | Стандарт формата планов. Без него планы теряют единообразие. |
| `docs/exec-plans/tech-debt-tracker.md` | Единое место для долга. |
| `scripts/verify.sh` | Главный инструмент проверки. |

> **Правило для агента:** Если задача требует удаления или переименования любого из этих файлов — ОСТАНОВИСЬ и запроси подтверждение у человека (escalate).

### B.2. Как работать с Exec Plans (Планы выполнения)

Exec Plans — это сердце Context Graph. Они сохраняют «почему мы сделали именно так».

**Создание плана:**
1. Скопируй `docs/exec-plans/TEMPLATE.md` в `docs/exec-plans/active/<имя-задачи>.md`.
2. Заполни секции `Context & Purpose` и `Execution Steps`.
3. Укажи в поле `Branch` имя своей git-ветки.

**Ведение плана (в процессе работы):**
1. Отмечай выполненные шаги: `- [x] Шаг. (YYYY-MM-DD HH:MM)`.
2. **ОБЯЗАТЕЛЬНО** записывай неожиданные находки в `Surprises & Discoveries`.
3. Для каждого сюрприза: если возможно — напиши тест или добавь проверку в линтер. Укажи ссылку на тест в поле `Harness Update`.
4. Если пришлось выбирать между подходами — зафиксируй выбор и причину в `Decision Trace`.

**Закрытие плана:**
1. Убедись, что `scripts/verify.sh` проходит без ошибок.
2. Перемести файл из `active/` в `completed/`.
3. Перенеси пункты из `Leftover Tech Debt` в `docs/exec-plans/tech-debt-tracker.md`.
4. Установи `Status: Completed`.

### B.3. Как работать с Design Docs (Проектные решения)

Design Docs описывают фундаментальные архитектурные решения. Они меняются редко и только через осознанный процесс.

**Создание:** Создай новый `.md` файл в `docs/design-docs/`. Добавь запись в `docs/design-docs/index.md` со статусом `Draft`.

**Жизненный цикл статусов:** `Draft` → `Active` → `Superseded` (заменён новым).

**Важно:** Если при работе над Exec Plan выяснилось, что текущий Design Doc устарел — **сначала обнови Design Doc**, потом пиши код. Не наоборот.

### B.4. Как работать с References (Справочные материалы)

Файлы в `docs/references/` — это шпаргалки, которые помогают агентам не галлюцинировать.

**Когда добавлять:** Когда агент повторно ошибается в использовании конкретной библиотеки, API или инструмента. Не добавляй «на всякий случай» — добавляй по факту реальной проблемы.

**Формат:** Свободный (`.md` или `.txt`). Обязательно включай конкретные примеры кода.

### B.5. Правило «Никаких сюрпризов без тестов» (Test-Driven Harness)

Если ты (агент) обнаружил неожиданное поведение (баг библиотеки, особенность API, подводный камень) и записал его в `Surprises & Discoveries`:

1. **Первый приоритет:** Напиши юнит-тест, фиксирующий это поведение.
2. **Если тест невозможен** (например, внешний API с rate limiting): добавь runtime-проверку (assertion) в код или кастомное правило линтера.
3. **В крайнем случае:** Добавь правило в `AGENTS.md` или `.cursor/rules/`, чтобы предотвратить повторение ошибки.

Цель: каждый сюрприз должен быть **механически защищён**, а не просто описан текстом.

### B.6. Harness Engineering (Инженерия среды)

Папка `scripts/` — это твой набор инструментов автоматической проверки.

**Главный принцип:** Каждый раз, когда агент допускает ошибку, которую можно обнаружить автоматически, — добавь проверку в `scripts/verify.sh` или создай отдельный скрипт.

**Контракт скрипта:** Каждый скрипт в `scripts/` должен быть описан в `scripts/index.md` с указанием: что делает, какие аргументы принимает, что возвращает (exit code, stdout).

### B.7. Что автоматизировать, а что оставить в тексте

| Тип правила | Где хранить | Пример |
|---|---|---|
| Проверяемое линтером | Конфиг линтера (`.eslintrc`, `ruff.toml` и т.п.) | «Не использовать `any` в TypeScript» |
| Проверяемое тестом | `tests/` | «API возвращает 429 при >5 req/s» |
| Проверяемое скриптом | `scripts/` | «Размер бандла не должен превышать 500KB» |
| Нелинтируемый принцип | `docs/design-docs/core-beliefs.md` | «Предпочитаем дублирование сложной абстракции» |
| Инструкция для агента | `AGENTS.md` или `.cursor/rules/` | «Не читай /completed/ целиком» |

## Часть C: Правила мульти-агентной работы

### C.1. Изоляция через ветки

- **Один Exec Plan = одна git-ветка = один агент.**
- Не редактируй файлы в `docs/exec-plans/active/`, которые относятся к чужому плану.
- Не используй `git stash` — другие агенты могут работать параллельно.

### C.2. Конфликты в глобальных документах

Если твоя задача требует изменения глобального файла (`ARCHITECTURE.md`, `core-beliefs.md`, `docs/index.md`):

1. Внеси изменения в своей ветке.
2. Перед мерджем выполни `git rebase main` (или `git merge main`).
3. Если возник конфликт — разреши его, сохранив логику обеих веток.
4. Задача не считается завершённой, пока глобальные изменения не влиты в `main`.

### C.3. Работа с папкой `completed/`

> **ЗАПРЕЩЕНО** читать папку `docs/exec-plans/completed/` целиком или перебирать все файлы в ней.

Если тебе нужен контекст из завершённых планов (например, почему код написан определённым образом):
- Используй **семантический поиск** (Semantic Search) или **grep** по `docs/exec-plans/completed/` с ключевыми словами (имена модулей, функций, технологий).
- Читай **только найденные релевантные файлы**, не все подряд.

## Часть D: Как расширять этот каркас

Этот каркас спроектирован для роста. Вот как добавлять новые элементы:

| Что хочешь добавить | Куда | Как |
|---|---|---|
| Новый модуль/домен | `src/<модуль>/` | Опционально добавь `AGENTS.md` внутри модуля с локальными правилами |
| Новую технологию в стек | `docs/references/` | Создай файл-шпаргалку и обнови `docs/references/index.md` |
| Новое архитектурное решение | `docs/design-docs/` | Создай Design Doc и обнови `docs/design-docs/index.md` |
| Новый скрипт проверки | `scripts/` | Создай скрипт и обнови `scripts/index.md` |
| Новое продуктовое требование | `docs/product-specs/` | Создай спецификацию и обнови `docs/product-specs/index.md` |
| Правило для IDE | `.cursor/rules/` | Создай или обнови `.mdc` файл |

**Главное правило расширения:** При добавлении нового файла в любую папку `/docs/` — обнови соответствующий `index.md`. Индексы должны быть актуальны всегда.
