# Архитектура проекта

> Этот документ описывает верхнеуровневую структуру системы: модули, слои, границы и допустимые зависимости.
> Обновляется только через Exec Plan (см. `docs/index.md`, секция B.3).

## Статус

**Текущий этап:** Каркас (scaffolding). Проект находится в стадии формирования структуры.
Конкретные модули, стек технологий и слои будут описаны при старте разработки.

## Общие архитектурные принципы

1. **Строгие границы между слоями.** Зависимости разрешены только «вперёд» (от верхних слоёв к нижним). Обратные зависимости запрещены.
2. **Parse, don't validate.** Данные на входе в любой слой парсятся в типизированные структуры, а не просто проверяются на «непустоту».
3. **Boring technology.** Предпочитаем стабильные, хорошо документированные технологии с широким представительством в обучающих данных LLM.
4. **Repository as single source of truth.** Всё, что агент должен знать, лежит в репозитории. Если решение принято в чате/Slack — оно должно быть зафиксировано в docs/.

## Структура модулей

> Заполняется при старте конкретного проекта. Шаблон:

```
src/
├── <модуль-A>/
│   ├── AGENTS.md          — (опционально) локальные правила для этого модуля
│   ├── types/             — типы и схемы данных
│   ├── services/          — бизнес-логика
│   └── ...
│
├── <модуль-B>/
│   └── ...
│
└── shared/                — общие утилиты и хелперы
```

## Диаграмма зависимостей

> Заполняется при появлении конкретных модулей. Формат:

```
Types → Config → Repository → Service → Runtime → UI
                                ↑
                      Providers (auth, telemetry, feature flags)
```

Каждый бизнес-домен строится из фиксированного набора слоёв.
Код может зависеть только «вперёд» по цепочке. Кросс-доменные зависимости идут только через Providers.
